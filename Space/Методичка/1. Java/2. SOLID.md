## Что такое SOLID 
**SOLID** – это набор пяти базовых принципов объектно-ориентированного
проектирования, помогающих писать легко поддерживаемый,
расширяемый и надежный код. Каждый принцип задает правило
архитектуры классов и модулей.
## 🛠Принцип единственной ответственности (Single Responsibility Principle)
**Краткое определение**: каждый класс должен иметь одну-единственную
ответственность (причину для изменения). Это означает, что класс
должен решать ровно одну задачу или управлять одной частью
функциональности системы.
📜 Пример нарушения и исправления: представим класс, который и
обрабатывает заказ, и отправляет email-уведомления клиенту. Это две
разные ответственности.

```java
// Нарушение SRP: OrderService обрабатывает заказ и отправляет уведомления
public class OrderService {
	public void processOrder(Order order) {
	// ... логика обработки заказа ...

		sendEmailNotification(order);
	}
	private void sendEmailNotification(Order order) {
		// код отправки Email (например, JavaMail)
		System.out.println("Отправка email для заказа #" +
		order.getId());
	}
}
```
В данном примере **OrderService** отвечает сразу за две задачи: обработку
заказа и отправку уведомления. Правильнее разделить эти обязанности:
```java
// Исправлено: разделили обязанности на два класса
public class OrderService {
	private EmailSender emailSender;
	public OrderService(EmailSender emailSender) {
		this.emailSender = emailSender;
	}
	public void processOrder(Order order) {
	// ... логика обработки заказа ...
		emailSender.sendEmail(order);
	}
}
public class EmailSender {
	public void sendEmail(Order order) {
		// код отправки Email
		System.out.println("Отправка email для заказа #" +
		order.getId());
	}
}
```
Теперь **OrderService** отвечает только за бизнес-логику заказа, а класс
EmailSender — за отправку электронной почты. Такой подход облегчает
поддержку кода и позволяет тестировать каждую часть отдельно.
## 🔒 Принцип открытости/закрытости (Open/Closed Principle, OCP)
**Краткое определение**: программные сущности (классы, модули, функции)
должны быть открыты для расширения, но закрыты для изменения. То есть поведение системы можно расширять (добавлять новую функциональность) без изменения уже написанного кода.

📜 Пример нарушения и исправления: представим класс, который
обрабатывает оплату разными способами на основе типа:
```java
// Нарушение OCP: добавление нового способа оплаты требует изменения класса
public class PaymentProcessor {
	public void processPayment(String type, double amount) {
		if ("CREDIT_CARD".equals(type)) {
		// обработка оплаты картой
		} else if ("PAYPAL".equals(type)) {
		// обработка PayPal
		}
		// Новые условия потребуют добавлять новые ветки if/else
	}
}
```
Такое решение закрыто для расширения: при появлении еще одного
типа платежа потребуется изменить код метода processPayment.
Исправление – использовать интерфейс и класс для каждого способа
оплаты (Стратегия):
```java
	// Интерфейс платежного метода
public interface PaymentStrategy {
	void pay(double amount);
}
// Реализация оплаты кредитной картой
public class CreditCardPayment implements PaymentStrategy {
	public void pay(double amount) {
		System.out.println("Оплата " + amount + " кредитной
		картой");
	}
}
// Реализация оплаты через PayPal
public class PayPalPayment implements PaymentStrategy {
	public void pay(double amount) {

		System.out.println("Оплата " + amount + " через PayPal");
	}
}
// Класс, использующий стратегию оплаты
public class OrderService {
	private PaymentStrategy paymentStrategy;
	public OrderService(PaymentStrategy paymentStrategy) {
		this.paymentStrategy = paymentStrategy;
	}
	public void processOrder(double amount) {
		paymentStrategy.pay(amount);
	}
}
```
Теперь, чтобы добавить новый способ оплаты (например,
**BitcoinPayment**), нужно написать новый класс, реализующий
PaymentStrategy, и передать его в OrderService. Никакие существующие
классы не меняются.
## 🌀Принцип подстановки Лисков (Liskov Substitution Principle, LSP)
**Краткое определение**: объекты подкласса должны быть
взаимозаменяемы с объектами базового класса, не нарушая
корректности программы. Проще говоря: если у нас есть код,
работающий с базовым типом, он должен одинаково корректно работать
с любым его подклассом.
**📜 Пример нарушения LSP**: класс **Rectangle** и его подкласс Square. У
прямоугольника можно независимо задать ширину и высоту. Квадрат
всегда имеет равную ширину и высоту. Если мы сделаем Square
наследником Rectangle и будем переопределять сеттеры так, чтобы
ширина и высота всегда совпадали, поведение изменится:
```java
// Нарушение LSP: подкласс Square изменяет контракт Rectangle
public class Rectangle {
	private int width;
	private int height;
	public void setWidth(int w) { this.width = w; }
	public void setHeight(int h) { this.height = h; }

	public int getWidth() { return width; }
	public int getHeight() { return height; }
	public int getArea() { return width * height; }
}
public class Square extends Rectangle {
	@Override
	public void setWidth(int w) {
		super.setWidth(w);
		super.setHeight(w);
}
	@Override
	public void setHeight(int h) {
		super.setHeight(h);
		super.setWidth(h);
	}
}
// Демонстрация ошибки подстановки:
Rectangle rect = new Rectangle();
rect.setWidth(5);
rect.setHeight(10);
System.out.println(rect.getArea()); // 50 - всё как ожидается
Rectangle sq = new Square();
sq.setWidth(5);
sq.setHeight(10);
System.out.println(sq.getArea()); // 100! А пользователь ожидал 50,
как в Rectangle
```
**В примере мы нарушили LSP**: код, работающий с Rectangle, ведет себя
неожиданно при передаче Square вместо Rectangle. Исправлять
подобное можно, избежав некорректного наследования (например, не
наследовать Square от Rectangle) или пересмотрев дизайн.
## 🔌 Принцип разделения интерфейсов (Interface Segregation Principle, ISP)
**Краткое определение**: клиенты не должны зависеть от методов
интерфейса, которые они не используют. Лучше иметь много
узкоспециализированных интерфейсов, чем один «толстый»
универсальный. Если интерфейс слишком большой, его следует
разделить на несколько маленьких.

**📜 Пример нарушения и исправления**: возьмем пример с работниками:
есть интерфейс с методами work() и eat(), который реализуют как люди,
так и роботы. Робот eat() не нужен, но ему приходится его реализовать:
```java
// Нарушение ISP: общий интерфейс Worker слишком большой
public interface Worker {
	void work();
	void eat();
}
public class HumanWorker implements Worker {
	public void work() { System.out.println("Человек работает"); }
	public void eat() { System.out.println("Человек ест"); }
}
public class RobotWorker implements Worker {
	public void work() { System.out.println("Робот работает"); }
	
	public void eat() {
		// у робота нет обеда, метод не нужен
		throw new UnsupportedOperationException("Робот не может
		есть");
	}
}
```
Интерфейс Worker слишком «толстый». Разделим его на более узкие:
```java
// Исправлено: разделили интерфейс на два
public interface Workable {
	void work();
}
public interface Feedable {
	void eat();
}
public class HumanWorker implements Workable, Feedable {
	public void work() { System.out.println("Человек работает"); }
	public void eat() { System.out.println("Человек ест"); }
}

public class RobotWorker implements Workable {
	public void work() { System.out.println("Робот работает"); }
	// нет метода eat(), и всё хорошо
}
```
Теперь RobotWorker реализует только Workable, без ненужного eat().
Интерфейсы стали более специализированы.
## 🤝 Принцип инверсии зависимостей (Dependency Inversion Principle, DIP)
**Краткое определение**: модули верхнего уровня не должны зависеть от
модулей нижнего уровня – оба должны зависеть от абстракций.
Абстракции не должны зависеть от деталей, детали должны зависеть от
абстракций. Проще говоря: вы должны работать через интерфейсы
(абстракции), а не через конкретные классы.
📜 Пример нарушения и исправления: допустим, есть сервис, который
отправляет уведомления клиенту. Он напрямую использует конкретный
класс EmailService:
```java
// Нарушение DIP: жесткая зависимость от конкретного класса
EmailService
public class OrderService {
	private EmailService emailService = new EmailService();
	public void processOrder(Order order) {
		// ... логика обработки заказа ...
		emailService.send(order);
	}
}
public class EmailService {
	public void send(Order order) {
		System.out.println("Отправка Email заказчику");
	}
}
```
Здесь OrderService зависит от конкретного EmailService. Что если
понадобится отправлять не только Email, но и SMS? Придётся менять
OrderService.
Правильный подход – зависеть от абстракции:
```java
// Абстракция (интерфейс для уведомлений)
public interface Notifier {
	void send(Order order);
}
// Email-реализация
public class EmailNotifier implements Notifier {
	public void send(Order order) {
		System.out.println("Отправка Email заказчику");
	}
}
// SMS-реализация
public class SmsNotifier implements Notifier {
	public void send(Order order) {
		System.out.println("Отправка SMS заказчику");
	}
}
// Сервис заказов с внедрением зависимости
public class OrderService {
	private Notifier notifier;
	public OrderService(Notifier notifier) {
		this.notifier = notifier;
	}
	public void processOrder(Order order) {
		// ... логика обработки заказа ...
		notifier.send(order);
	}
}
```
Теперь OrderService работает с интерфейсом Notifier. Можно передать
любую реализацию (EmailNotifier, SmsNotifier и т.д.). Это соответствует DIP:
высокоуровневый модуль (OrderService) не зависит от деталей
(EmailNotifier).
## 🪄 Когда сознательно можно нарушить DIP: иногда ради простоты и
экономии времени. Например:
🔹 В очень простых скриптах или при прототипировании, где гибкость не
нужна.
🔹 В маленьких утилитах или очень стабильном коде, который вряд ли
будет меняться.
🔹 Когда введение абстракций принесет больше сложности, чем пользы
(принцип YAGNI – «не привносить лишнего»).
