## Что такое SOLID 
**SOLID** – это набор пяти базовых принципов объектно-ориентированного
проектирования, помогающих писать легко поддерживаемый,
расширяемый и надежный код. Каждый принцип задает правило
архитектуры классов и модулей.
## Принцип единственной ответственности (Single Responsibility Principle)
**Краткое определение**: каждый класс должен иметь одну-единственную
ответственность (причину для изменения). Это означает, что класс
должен решать ровно одну задачу или управлять одной частью
функциональности системы.
📜 Пример нарушения и исправления: представим класс, который и
обрабатывает заказ, и отправляет email-уведомления клиенту. Это две
разные ответственности.

```java
// Нарушение SRP: OrderService обрабатывает заказ и отправляет уведомления
public class OrderService {
	public void processOrder(Order order) {
	// ... логика обработки заказа ...

		sendEmailNotification(order);
	}
	private void sendEmailNotification(Order order) {
		// код отправки Email (например, JavaMail)
		System.out.println("Отправка email для заказа #" +
		order.getId());
	}
}
```
В данном примере **OrderService** отвечает сразу за две задачи: обработку
заказа и отправку уведомления. Правильнее разделить эти обязанности:
```java
// Исправлено: разделили обязанности на два класса
public class OrderService {
	private EmailSender emailSender;
	public OrderService(EmailSender emailSender) {
		this.emailSender = emailSender;
	}
	public void processOrder(Order order) {
	// ... логика обработки заказа ...
		emailSender.sendEmail(order);
	}
}
public class EmailSender {
	public void sendEmail(Order order) {
		// код отправки Email
		System.out.println("Отправка email для заказа #" +
		order.getId());
	}
}
```
Теперь **OrderService** отвечает только за бизнес-логику заказа, а класс
EmailSender — за отправку электронной почты. Такой подход облегчает
поддержку кода и позволяет тестировать каждую часть отдельно.
## 🔒 Принцип открытости/закрытости (Open/Closed Principle, OCP)
**Краткое определение**: программные сущности (классы, модули, функции)
должны быть открыты для расширения, но закрыты для изменения. То есть поведение системы можно расширять (добавлять новую функциональность) без изменения уже написанного кода.

📜 Пример нарушения и исправления: представим класс, который
обрабатывает оплату разными способами на основе типа:
```java
// Нарушение OCP: добавление нового способа оплаты требует изменения класса
public class PaymentProcessor {
	public void processPayment(String type, double amount) {
		if ("CREDIT_CARD".equals(type)) {
		// обработка оплаты картой
		} else if ("PAYPAL".equals(type)) {
		// обработка PayPal
		}
		// Новые условия потребуют добавлять новые ветки if/else
	}
}
```
Такое решение закрыто для расширения: при появлении еще одного
типа платежа потребуется изменить код метода processPayment.
Исправление – использовать интерфейс и класс для каждого способа
оплаты (Стратегия):
```java
	// Интерфейс платежного метода
public interface PaymentStrategy {
	void pay(double amount);
}
// Реализация оплаты кредитной картой
public class CreditCardPayment implements PaymentStrategy {
	public void pay(double amount) {
		System.out.println("Оплата " + amount + " кредитной
		картой");
	}
}
// Реализация оплаты через PayPal
public class PayPalPayment implements PaymentStrategy {
	public void pay(double amount) {

		System.out.println("Оплата " + amount + " через PayPal");
	}
}
// Класс, использующий стратегию оплаты
public class OrderService {
	private PaymentStrategy paymentStrategy;
	public OrderService(PaymentStrategy paymentStrategy) {
		this.paymentStrategy = paymentStrategy;
	}
	public void processOrder(double amount) {
		paymentStrategy.pay(amount);
	}
}
```
Теперь, чтобы добавить новый способ оплаты (например,
BitcoinPayment), нужно написать новый класс, реализующий
PaymentStrategy, и передать его в OrderService. Никакие существующие
классы не меняются.
